> Ok, but what if have to debug dbutils.c for example? It is not a parser
> file, but i need to parse it anyway. I have to compile it with debug option
> and also able to run it as a "standalone" entity, (I don't believe is
> possible). So, I have to compile the entire SN for debugging that single
> file?

Yes, you should configure SN with

#> CFLAGS="-O2 -g" ./configure --prefix=/some/thing

so the all object files get debug symbols; this needs to be done due
to the fact that all executables link to various libs (db, tcl,
dbutils, etc).

To debug parsers there's somewhere in the source a script called
"snpdbg", which can help you debugging parsers. Nontheless it's still
rather complicated.
I usually do it by invoking a parser manually (the options are display
when running snavigator -debug 99 for example).

SN's debug facilities aren't very good, I have to admit that.

Also to Earnie: As for using RamDebugger - I think it's rather
complicated to the fact that "hyper" is adding new tcl/tk functions to
the tcl interpreter which are not readily available via a tcl script,
but are purely implemented in C.
