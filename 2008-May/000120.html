I've just committed my changes to the input configuration files
(Makfile.am, configure.in, *.m4).  autoconf 2.61 and automake 1.10
were used to create user configuration files (Makfile.in, configure,
etc).

SN will now (once again) configure and build in a separate build
directory (ie out of the source tree).
db4 now builds a db1.85 compatible version by default.
The db4 header files needed for the SDK are now accessed via the build tree.
There was one change required in a source file
(tcl/unix/tclUnixFCmd.c) to handle a change-of-name of a #define item
due to upgrade of autoconf 2.13->2.61

A bug in a config/acinclude.m4 macro to find Tcl/Tk headers as been
corrected.  This came to light because the new configure scripts set
the cache file to /dev/null by default (ie no cache).  Previous
behaviour did use a cache file which masked the m4 bug.

The configuration, build, install, smoke-test cycle has been done on
Mac OS X (Leopard).  Further testing on other platforms would be
appreciated (I can/will do Solaris 8).

I will write a cookbook to document how to use autoreconf across the
project Real Soon Now.
