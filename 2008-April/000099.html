Hi Daniel.

On Sun, 2008-04-27 at 20:57 +0200, Daniel Scheibli wrote:
> I'am using SN for a quite large project (ca. 37K files) -
> mostly doing full text scans over the entire code base. In
> the best case scenario such a run takes about 12 seconds,
> but this requires that all source files are already present
> in the file system cache. If this is not the case, then we
> are talking about a few minutes runtime.

In my case i was talking more about the time it actually takes to open
the source file when i click it (from the list of files found during
grep). But of-course the grep itself could also be improved. 

> The question is how to improve that. I think one key
> question to answer would be how up to date such a search
> has to be? Are people using SN for searching a "static"
> snapshot of a given software (say Linux kernel source
> A.B.C), or are they using SN to process code they are
> actively working on (i.e. more as a IDE). I myself use
> SN in the first context only, but I'am curios how others
> are using SN.

Personally i use it for both. I often re-base the code via git, cvs or svn,
and then search again with SN-grep.

>From my point of view the point about TCL/TK is important. If others are going 
to contribute with patches and code, i think it is a show-stopper that
SN still uses these old versions. The problem is that most programmers
do not want to work with old versions of anything. Lets say that want to 
improve SN by fixing a bug in TCL or TK

Normally you would have to:

1) Convince TCL/TK devs to fix it (or send patch)....
2) Wait for automatic update in your distro (or upgrade manually).

But here you have to:

1) Convince TCL/TK devs to fix it (or send patch)....
2) Backport fix to older version, which could give some headaches.
3) Convince SN devs to include fix in their special version.

The second path is not desirable for many reasons. One good reason is that
it gives poor quality, as only the small team using SN gets to verify
and test the changes. If its changed on TCL/TK mainline it is theoretically 
verified by millions.

/Peter
