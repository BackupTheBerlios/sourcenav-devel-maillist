Quoting Mark Thornber <mark.thornber at gmail.com>:

> 2008/4/28 Peter Dons Tychsen <donpedro at tdcadsl.dk>:
>>  >From my point of view the point about TCL/TK is important. If 
>> others are going
>>  to contribute with patches and code, i think it is a show-stopper that
>>  SN still uses these old versions. The problem is that most programmers
>>  do not want to work with old versions of anything. Lets say that want to
>>  improve SN by fixing a bug in TCL or TK
> I agree with the principle of having dependencies on packages provided
> by others - as long as there is a good package manager available - and
> all target platforms provide the same level of functionality.
>
> However - we are where we are due to historical reasons.  I started
> using SN in 1998 (before it went open source) and I suspect the idea
> of bundling Tcl/Tk etc was to provide an easy installation -
> especially on Windows.  10 years on and package managers have improved
> but I'm still not sure that there is an easy way to install the
> dependencies on a Windows platform.
>

I'll suggest Cygwin's setup.exe which is FOSS.  It may need to be 
modified some to fit but dependency checking is already programmed.  It 
is all controlled via setup.ini and setup.hint for the dependency 
package.

> Also, AFAIKR, Tcl/Tk was 'fluid' at that time and having Tcl/Tk
> bundled in would have made SN more stable and insulated from the
> changes happening in the Tcl/Tk world.  Another consideration is
> whether the original developers tweaked the Tcl/Tk distribution for
> their own reasons.
>

I think it was both reasons but primarily the latter.  I remember 
trying a long while ago breaking out tcl/tk and other pieces but didn't 
get very far (I didn't have the time to spend).

> Having said all that we do want to move forward to later versions of
> Tcl/Tk but we felt that getting a stable backend database was top of
> the list.
>

Yes, that would be more important.  Another decision that might need to 
be considered is do we stick with TCL/TK or move to say Ruby, Python, 
or something else.  Both Ruby and Python could provide net appliances 
that could be useful for collaborative work.
