Hi Peter,

> 5) Opening files in "grep" is fairly slow, even on my 3GHz
> machine. Could this somehow be optimized? I understand the
> search takes time, but sometimes opening the file takes
> longer than the search! :-)

I second that observation.

I'am using SN for a quite large project (ca. 37K files) -
mostly doing full text scans over the entire code base. In
the best case scenario such a run takes about 12 seconds,
but this requires that all source files are already present
in the file system cache. If this is not the case, then we
are talking about a few minutes runtime.

The question is how to improve that. I think one key
question to answer would be how up to date such a search
has to be? Are people using SN for searching a "static"
snapshot of a given software (say Linux kernel source
A.B.C), or are they using SN to process code they are
actively working on (i.e. more as a IDE). I myself use
SN in the first context only, but I'am curios how others
are using SN.

-Daniel
